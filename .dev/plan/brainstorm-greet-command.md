# brainstorm: greet-command

## 確定事項

1. **実装言語**: bash（ユーザー指定）。shebang `#!/usr/bin/env bash`、`chmod +x`。
2. **配置**: 単一ファイル `src/greet`（サブコマンドなし）。
3. **出力形式**: plain text のみ（例: `Hello, Alice!`）。JSON や `--format` は採用しない。理由: テストテーマ（brainstorm 結論明示 / Edit 範囲ガイドライン / 双方向リンク可読性）の検証素材として出力形式は濃すぎるため、最小に倒す。
4. **名前未指定時のフォールバック**: エラー終了（exit 1 + stderr に usage）。Issue 本文に明示はないが CLI 慣習として素直。既定値 "World" は採用しない。
5. **引数パース方式**: 自前 while-case パース。`--lang <code>` / `--help` / `-h` / 位置引数 1 個（名前）。`getopts` は長オプション非対応のため不採用。
6. **CLI インタフェース**:
   - `greet <name>` で挨拶を出力
   - `--lang en|ja` で言語選択（デフォルト `en`）
   - `--help` / `-h` で usage 表示、exit 0
   - 終了コード: 成功 0 / 引数不正 1 / 未対応言語は要検討事項1 参照
7. **テスト形式**: 手動実行手順ベース（`test_cmd: ls test-specs/*-test-spec.md` はファイル存在確認のみのため、テスト仕様書本体に手動手順を記載）。
8. **多言語対応**: `--lang en|ja` の 2 言語対応を採用。理由: テストテーマ 1（Edit 範囲ガイドラインの「軽微 vs 仕様変更」境界）の素材として「言語 1 個追加」が典型的な境界事例になるため、意図的に拡張余地を残す。

## 要検討事項

1. **未対応言語指定時の挙動**
   - 案A: エラー終了（exit 2、stderr に `unsupported language: xx` を出力） — CLI 慣習的に素直。ユーザーの誤入力を早期に気づかせる
   - 案B: 英語フォールバック（警告なしで en として動作） — 寛容だが誤入力がサイレントに通る
   - 案C: stderr に警告を出しつつ英語で出力、exit 0 — 両取りだが挙動が曖昧
   - 推奨: **案A**。Issue 本文に明示はないが、CLI の慣習からは案 A が素直。ただし DESIGN フェーズで ASSUMPTION マーカーを付けて最終確認する価値あり（テストテーマ 2「結論明示ガイドライン」の検証素材）。

2. **対応言語の拡張方式（実装内部構造）**
   - 案A: case 文ハードコード（`case "$lang" in en) ...;; ja) ...;; esac`） — 最小・直感的
   - 案B: bash 連想配列（`declare -A GREETINGS=([en]="Hello" [ja]="こんにちは")`） — bash 4+ 依存、拡張時は 1 行追加で済む
   - 案C: 外部ファイル（`src/greet.i18n.sh` を source） — オーバースペック
   - 推奨: **案A（case 文）**。プロジェクトの性質（デバッグ題材・最小構成）から案 A が自然。この判断は「言語 1 個追加が軽微な書き換えか仕様変更か」の境界テスト（テストテーマ 1）の素材として DESIGN に残す価値あり。

## リスク・懸念

1. **テストテーマと実装題材の乖離リスク**: greet 自体は極めて単純なため、設計書・テスト仕様書が薄くなりすぎると各フェーズの Agent 挙動検証が不十分になる可能性。→ 要検討事項 1・2 を DESIGN に残すことで最低限の判断量を確保する。
2. **bash 4+ 前提の暗黙化**: 連想配列等を使わない方針（要検討事項2 推奨案A）なら bash 3.2 でも動くが、明記しないと環境依存バグの温床に。→ DESIGN で bash バージョン要件を明記する。
3. **ダミー build/check/test コマンドの実質性欠如**: `ls src/*` 等はファイル存在確認のみで、実装バグは検出できない。→ 本プロジェクトの目的（/develop スキルのデバッグ）上は許容。テスト仕様書に手動手順を記載することで補う。

## 設計入力

### 外部仕様・依存
- bash（バージョンは DESIGN で明記。要検討事項2 の推奨案Aを採れば bash 3.2+、案Bなら 4.0+）
- POSIX 標準コマンドのみ（`echo`, `printf` 程度）
- 外部ライブラリ・パッケージマネージャなし

### スコープ境界
- やること:
  - `src/greet` 単一ファイルの bash CLI 実装
  - `--lang en|ja` / `--help` / 位置引数 1 個（名前）
  - `designs/` に 1 本の設計書
  - `test-specs/` に 1 本のテスト仕様書（手動実行手順）
- やらないこと:
  - JSON 出力、`--format` オプション
  - 3 言語以上への対応（拡張余地は残すが本スコープ外）
  - 自動テスト（unit/integration test）
  - サブコマンド、対話モード、設定ファイル

### 検証方針
- 静的検証: `check_cmd: ls designs/*.md` でファイル存在確認
- 自動テスト: `test_cmd: ls test-specs/*-test-spec.md` でファイル存在確認のみ（実質ダミー）
- ビルド: `build_cmd: ls src/*` でファイル存在確認
- 手動検証: テスト仕様書に記載する以下のケースを実行し目視確認
  - 正常系: `./src/greet Alice` → `Hello, Alice!`
  - 日本語: `./src/greet --lang ja Alice` → `こんにちは、Alice！`
  - ヘルプ: `./src/greet --help` → usage 表示、exit 0
  - 名前未指定: `./src/greet` → stderr に usage、exit 1
  - 未対応言語: `./src/greet --lang fr Alice` → 要検討事項1 の確定後に記載

### データモデル差分
- 該当なし（永続データなし）

### 後方互換・フォールバック
- 新規追加のため後方互換の考慮なし

### バージョニング方針
- 該当なし（本プロジェクトはバージョニング対象外のデバッグ題材）

## 未深堀り論点

1. **ログ出力先の規約**（stdout vs stderr）: 正常出力は stdout、エラー・usage は stderr という慣習で自明のため深堀り省略。DESIGN で usage の出力先のみ明記する。
2. **国際化（i18n）の文字エンコーディング**: UTF-8 前提で自明のため省略。ja 出力に非 ASCII が含まれるため `LANG`/`LC_ALL` の扱いは DESIGN で念のため触れる。
